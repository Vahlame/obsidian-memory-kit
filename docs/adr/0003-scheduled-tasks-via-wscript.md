# ADR-0003: Run scheduled tasks via `wscript.exe` and a VBS shim

- **Status:** Accepted
- **Date:** 2026-05-13
- **Public guide scope (2026-05):** end-user docs no longer ship VBS/PowerShell task templates; prefer **stdio** `basic-memory`, **`obsidian-memoryd`**, or **self-defined** Task Scheduler actions. This ADR documents the historical pattern for private v1 vaults and rationale.

## Context

Both scheduled tasks (`CursorObsidianMcpWatchdog`, `CursorMemoryAutoSync`) run frequently (every 5 and 10 minutes). If they are wired to invoke `powershell.exe` directly, Windows shows a console window every single time the task fires. That is unacceptable UX.

PowerShell does have `-WindowStyle Hidden`, but Task Scheduler still flashes a console frame on launch.

## Decision

The task points at `wscript.exe //B //nologo <runner>.vbs`. The VBS shim uses `WScript.Shell.Run` with the third argument set to `0` (hidden window). The VBS then launches PowerShell, which executes the actual `.ps1`. End user sees no console at all.

```vbscript
Set shell = CreateObject("WScript.Shell")
shell.Run "powershell.exe -NoProfile -NonInteractive -ExecutionPolicy Bypass -File ""...ps1"" -VaultPath ""...""", 0, True
```

## Alternatives considered

- **`powershell.exe -WindowStyle Hidden`.** Flashes a window on cold launch. Visible noise on a 5-minute cadence is intolerable.
- **`pwsh.exe` via NSSM or sc.exe service.** Requires admin and a service install. Out of scope for this no-admin design.
- **`cmd /c start /B` wrapper.** Inconsistent: the console window still appears on some Windows builds.

## Consequences

- **Positive:** Tasks fire silently. The user never sees a console window.
- **Negative:** One extra file per task on disk (`run-watchdog-hidden.vbs`, `run-sync-hidden.vbs`). They live inside the vault and are auto-regenerated by the enable scripts.
- **Neutral:** VBS as a technology is in maintenance mode at Microsoft but it remains supported on all current Windows versions.
